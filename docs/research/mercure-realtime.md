# Mercure Real-Time Updates — Research

**Date:** 2026-04-06
**Context:** Issue #114 (async two-phase enrichment) will create articles first, then enrich asynchronously. Mercure would push updates to the browser when enrichment completes.

---

## FrankenPHP + Mercure Overview

FrankenPHP ships with a **built-in Mercure hub** — no separate container or service needed. The hub is embedded in the Caddy server that FrankenPHP extends.

### Current State in This Project

The infrastructure is **already configured** but the Symfony application does not use it yet:

- **Caddyfile** (`frankenphp/Caddyfile`): Mercure block is present with `publisher_jwt`, `subscriber_jwt`, `anonymous` subscribers, and `subscriptions` API enabled.
- **compose.yaml**: Environment variables are set — `MERCURE_URL`, `MERCURE_PUBLIC_URL`, `MERCURE_JWT_SECRET`, `MERCURE_PUBLISHER_JWT_KEY`, `MERCURE_SUBSCRIBER_JWT_KEY`.
- **compose.override.yaml**: Dev mode includes `MERCURE_EXTRA_DIRECTIVES: demo` (enables the Mercure debug UI at `/.well-known/mercure/ui`).
- **.env**: No Mercure variables yet (they come from compose environment).

The hub is available at `/.well-known/mercure` on the same origin as the app — no CORS issues.

### How It Works

1. **Publishing**: PHP code sends POST requests to the hub's internal URL (`MERCURE_URL`), authenticated with a JWT signed using `MERCURE_JWT_SECRET`.
2. **Subscribing**: Browser JavaScript opens an `EventSource` (SSE) connection to the hub's public URL (`MERCURE_PUBLIC_URL`). For public topics, no auth needed (anonymous mode is enabled). For private topics, a JWT cookie is used.
3. **Transport**: Uses Server-Sent Events (SSE) — native browser support, no WebSocket complexity, automatic reconnection, works through HTTP/2.

### FrankenPHP `mercure_publish()` (v1.10+)

FrankenPHP 1.10+ provides a native `mercure_publish()` PHP function that publishes directly to the built-in hub without an HTTP round-trip. However, the Symfony Mercure component uses the standard HTTP API, which is fine for this use case and more portable.

## Symfony Integration

### Installation

```bash
composer require symfony/mercure-bundle
```

This installs `symfony/mercure` (component) and `symfony/mercure-bundle` (framework integration). The Flex recipe creates `config/packages/mercure.php` (or `.yaml`).

### Configuration (`config/packages/mercure.php`)

```php
<?php

declare(strict_types=1);

use Symfony\Component\DependencyInjection\Loader\Configurator\ContainerConfigurator;

return static function (ContainerConfigurator $container): void {
    $container->extension('mercure', [
        'hubs' => [
            'default' => [
                'url' => '%env(MERCURE_URL)%',
                'public_url' => '%env(MERCURE_PUBLIC_URL)%',
                'jwt' => [
                    'secret' => '%env(MERCURE_JWT_SECRET)%',
                    'publish' => ['*'],
                ],
            ],
        ],
    ]);
};
```

The `publish: ['*']` grants permission to publish to all topics. The JWT is auto-generated by the bundle from the secret.

### Environment Variables (`.env`)

```dotenv
###> symfony/mercure-bundle ###
MERCURE_URL=http://php/.well-known/mercure
MERCURE_PUBLIC_URL=https://localhost:8443/.well-known/mercure
MERCURE_JWT_SECRET=!ChangeThisMercureHubJWTSecretKey!
###< symfony/mercure-bundle ###
```

These are already set in `compose.yaml` and override `.env` at runtime. Adding them to `.env` provides defaults for local/test environments.

### Publishing from PHP

```php
use Symfony\Component\Mercure\HubInterface;
use Symfony\Component\Mercure\Update;

final readonly class ArticleEnrichmentCompletePublisher
{
    public function __construct(
        private HubInterface $hub,
    ) {
    }

    public function publishEnrichmentComplete(int $articleId, array $data): void
    {
        $update = new Update(
            topics: [
                "/articles/{$articleId}/enrichment",
                '/articles',
            ],
            data: json_encode([
                'type' => 'enrichment_complete',
                'articleId' => $articleId,
                ...$data,
            ], JSON_THROW_ON_ERROR),
        );

        $this->hub->publish($update);
    }
}
```

Key points:
- **`HubInterface`** is auto-wired by the bundle.
- An `Update` takes topic(s), JSON data, and an optional `private` flag.
- Topics are strings (URIs by convention, but any string works).
- Multiple topics on one update = subscribers to any of them receive it.

### Publishing from a Messenger Handler

The enrichment handler (which runs in the worker process) can inject `HubInterface` and publish after enrichment completes. The worker container needs the `MERCURE_URL` and `MERCURE_JWT_SECRET` environment variables — **the worker currently does not have these** (see Docker Changes section below).

```php
#[AsMessageHandler]
final readonly class EnrichArticleHandler
{
    public function __construct(
        private ArticleEnrichmentServiceInterface $enrichment,
        private HubInterface $hub,
    ) {
    }

    public function __invoke(EnrichArticleMessage $message): void
    {
        $article = /* ... fetch and enrich ... */;

        $this->hub->publish(new Update(
            '/articles',
            json_encode([
                'type' => 'enrichment_complete',
                'articleId' => $article->getId(),
                'category' => $article->getCategory()?->getName(),
                'summary' => $article->getSummary(),
                'keywords' => $article->getKeywords(),
            ], JSON_THROW_ON_ERROR),
        ));
    }
}
```

## Use Cases & Topics

### Topic Design

| Event | Topic | Payload | Trigger |
|-------|-------|---------|---------|
| New article fetched | `/articles` | `{type: "new_article", articleId, title, sourceId}` | `FetchSourceHandler` after persist |
| Enrichment complete | `/articles/{id}/enrichment` | `{type: "enrichment_complete", articleId, category, summary, keywords}` | Enrichment handler after AI completes |
| Enrichment complete (broadcast) | `/articles` | Same as above | Same — dual-topic publish |
| Digest ready | `/digests` | `{type: "digest_ready", digestId}` | `GenerateDigestHandler` |
| Alert match | `/notifications` | `{type: "alert_match", ruleId, articleId}` | `SendNotificationHandler` |

### Priority Use Cases (for issue #114)

1. **Article enrichment complete**: The dashboard shows a skeleton/loading state for unenriched articles. When enrichment finishes, the browser receives an SSE event and fills in category, summary, and keywords without page reload.
2. **New article appears**: The dashboard shows a "N new articles" toast/banner. Click to prepend them.

### Secondary Use Cases (future)

3. **Digest generation complete**: Update the digest history page in real-time.
4. **Browser notifications**: Flash-style toast for alert matches.

## Frontend Implementation

### Option A: Vanilla EventSource (Recommended)

A standalone TypeScript module in `assets/ts/`, consistent with the existing pattern (`mark-as-read.ts`, `infinite-scroll.ts`, etc.). No framework dependency.

```typescript
// assets/ts/mercure-updates.ts

interface MercureConfig {
    hubUrl: string;
    topics: string[];
}

function buildSubscriptionUrl(config: MercureConfig): string {
    const url = new URL(config.hubUrl);
    for (const topic of config.topics) {
        url.searchParams.append('topic', topic);
    }
    return url.toString();
}

function connect(config: MercureConfig): EventSource {
    const url = buildSubscriptionUrl(config);
    const eventSource = new EventSource(url);

    eventSource.onmessage = (event: MessageEvent) => {
        const data = JSON.parse(event.data);
        handleUpdate(data);
    };

    eventSource.onerror = () => {
        // EventSource auto-reconnects; log for debugging
        console.warn('Mercure connection lost, reconnecting...');
    };

    return eventSource;
}

function handleUpdate(data: Record<string, unknown>): void {
    switch (data.type) {
        case 'new_article':
            showNewArticleBanner();
            break;
        case 'enrichment_complete':
            updateArticleCard(data.articleId as number, data);
            break;
        case 'digest_ready':
            showToast('New digest available');
            break;
    }
}

function showNewArticleBanner(): void {
    // Increment counter, show "N new articles" banner
    const banner = document.getElementById('new-articles-banner');
    if (banner) {
        const count = parseInt(banner.dataset.count ?? '0', 10) + 1;
        banner.dataset.count = String(count);
        banner.textContent = `${count} new article${count > 1 ? 's' : ''} available`;
        banner.classList.remove('hidden');
    }
}

function updateArticleCard(articleId: number, data: Record<string, unknown>): void {
    const card = document.querySelector(`[data-article-id="${articleId}"]`);
    if (!card) return;

    // Update category badge
    const categoryEl = card.querySelector('.article-category');
    if (categoryEl && data.category) {
        categoryEl.textContent = data.category as string;
        categoryEl.classList.remove('hidden');
    }

    // Update summary
    const summaryEl = card.querySelector('.article-summary');
    if (summaryEl && data.summary) {
        summaryEl.textContent = data.summary as string;
    }

    // Remove loading skeleton
    card.querySelector('.enrichment-loading')?.classList.add('hidden');
}

function showToast(message: string): void {
    // Use DaisyUI toast component
    const toast = document.createElement('div');
    toast.className = 'toast toast-end';
    toast.innerHTML = `<div class="alert alert-info"><span>${message}</span></div>`;
    document.body.appendChild(toast);
    setTimeout(() => toast.remove(), 5000);
}

// Initialize from data attributes on the body or a meta tag
const hubUrl = document.querySelector<HTMLMetaElement>('meta[name="mercure-hub"]')?.content;
if (hubUrl) {
    connect({
        hubUrl,
        topics: ['/articles', '/digests', '/notifications'],
    });
}
```

### Twig Integration

Add a meta tag in `base.html.twig` to pass the hub URL to JavaScript:

```html+twig
{% if app.user %}
    <meta name="mercure-hub" content="{{ mercure(['/articles', '/digests', '/notifications'])|escape('html_attr') }}">
{% endif %}
```

The `mercure()` Twig function (provided by the bundle) generates the full subscription URL including the hub URL and topic parameters.

Not recommended for this project — it does not use Stimulus anywhere else, and adding it just for Mercure would be over-engineering.

## Architecture & Config

### Docker Compose Changes

The **worker** service needs Mercure environment variables to publish updates:

```yaml
# compose.override.yaml (dev)
worker:
    environment:
        MERCURE_URL: http://php/.well-known/mercure
        MERCURE_JWT_SECRET: ${CADDY_MERCURE_JWT_SECRET:-!ChangeThisMercureHubJWTSecretKey!}

# compose.prod.yaml
worker:
    environment:
        MERCURE_URL: http://php/.well-known/mercure
        MERCURE_JWT_SECRET: ${CADDY_MERCURE_JWT_SECRET}
```

The worker publishes to the hub via `MERCURE_URL` (internal network). It does not need `MERCURE_PUBLIC_URL` (that's for browsers).

### Authentication Model

| Actor | Auth Method | Config |
|-------|-------------|--------|
| PHP publisher (app/worker) | JWT in Authorization header, auto-generated by bundle from `MERCURE_JWT_SECRET` | Bundle config `jwt.secret` + `jwt.publish: ['*']` |
| Browser subscriber (public topics) | None — anonymous mode enabled | Caddyfile `anonymous` directive (already present) |
| Browser subscriber (private topics) | JWT cookie set by Symfony | Requires `mercure_authorization` cookie — not needed for MVP |

For the initial implementation, **all topics are public** (anonymous subscription). This is appropriate because:
- Article data is not sensitive (it comes from public RSS feeds).
- The app is single-user / small-team.
- Private topics can be added later if multi-user access control is needed.

### Integration Points with Issue #114 (Two-Phase Enrichment)

Issue #114 splits article processing into:
1. **Phase 1**: Fetch + persist article (fast, synchronous within fetch handler)
2. **Phase 2**: Enrich article (async, via Messenger)

Mercure fits naturally:
- After Phase 1: Publish `new_article` event (dashboard shows article skeleton)
- After Phase 2: Publish `enrichment_complete` event (dashboard fills in AI-generated fields)

The `ArticleCreated` event already exists and is dispatched from `FetchSourceHandler::dispatchArticleEvents()`. A new `EventSubscriber` can listen for this event and publish a Mercure update — no changes to the handler itself.

For enrichment, the new async enrichment handler (created by #114) publishes a Mercure update after enrichment completes.

### No New Containers

FrankenPHP's built-in Mercure hub means:
- No separate Mercure container in docker-compose
- No additional port mapping (hub shares the app's HTTPS port)
- No additional TLS certificate management
- Hub scales with the app server

## Recommendation

**Go ahead with Mercure.** The infrastructure is already in place — this is low-hanging fruit.

### Why Mercure Is the Right Choice

1. **Zero infrastructure cost**: FrankenPHP already runs the hub. No new containers, ports, or services.
2. **Native Symfony integration**: `symfony/mercure-bundle` provides `HubInterface`, Twig helpers, and auto-configuration.
3. **SSE > WebSockets**: Simpler protocol, auto-reconnect, works through proxies, no bidirectional channel needed (this is a push-only use case).
4. **Perfect fit for #114**: Two-phase enrichment needs a way to push enrichment results to the browser. Without Mercure, the only option is polling.
5. **Small scope**: ~3-5 hours total including tests.

### Implementation Order

1. Install bundle + configure (can be done independently of #114)
2. Implement #114 (two-phase enrichment with Messenger)
3. Wire Mercure publishing into the enrichment pipeline
4. Add frontend subscription JS

### What to Avoid

- **Private topics for MVP**: Anonymous subscription is sufficient. Adding JWT cookies for subscribers adds complexity with no benefit for a single-user app.
- **Stimulus/Turbo**: The app uses vanilla TS. Adding Stimulus just for Mercure would be inconsistent.
- **Over-granular topics**: Start with `/articles` (catch-all) and `/digests`. Per-article topics (`/articles/{id}/enrichment`) can be added later if needed for targeted updates.
